## What this platform is

A commercial, industry-agnostic IoT platform (per `docs/` charter): device management → MQTT telemetry → rules engine → notifications → dashboards, designed for independent single-tenant deployments (one Postgres/Redis/EMQX/MinIO stack per customer). Core platform carries no industry-specific logic — verticals are meant to be added via device templates, rules, dashboards, and (as of Phase 12) **plugins**, not code changes to core.

## Stack

- **Backend**: Node.js (ESM), Express, PostgreSQL (`pg`), Redis (`ioredis`), EMQX via `mqtt` (MQTT.js), `nodemailer` (SMTP), `pdfkit`/`exceljs` (report export, Phase 10), MinIO via `minio` + `multer` (firmware/log file storage, Phase 11), a plugin framework (`src/plugins/`, Phase 12), `prom-client` + `compression` (production hardening, Phase 13), Joi validation, Pino logging, JWT auth, Jest + Supertest.
- **Frontend** (new in Phase 9): React + Vite SPA in `frontend/` (separate npm project), Recharts (charts/gauges), `react-leaflet`/Leaflet (maps). No frontend test framework yet.
- **Dev infra** (`docker-compose.dev.yml`): Postgres 16 (host port **5433**, not 5432 — see below), Redis 7 (6379), EMQX 5.7 (1883/18083), MinIO (9000/9001, **real as of Phase 11**), Mailpit (1025 SMTP / 8025 web+API). **Optional monitoring stack** (Phase 13, `docker-compose.monitoring.yml`): Prometheus (9090), Grafana (3300).
- **Testing philosophy**: no mocking of infrastructure — every test suite runs against the real containers above. This was a deliberate, consistently-applied choice across all phases.

### Phase 5 — MQTT Telemetry Platform (remediation)
Discovered that everything from "Phase 2" onward was non-functional facades (in-memory `Map`s standing in for Postgres, a fake `MqttClient` that never touched EMQX, no auth applied to any routes). Rebuilt for real: `pg.Pool`/`ioredis`/MQTT.js clients, idempotent SQL migrations (`001`-`003`), real telemetry ingestion pipeline (MQTT subscriber → Joi validation → Redis queue → worker → Postgres `telemetry_history` + Redis latest-value cache + `device_status`), offline detection (interval sweep), and auth-gating applied to all routes for the first time. Found and fixed a related pre-existing bug: `POST /auth/api-keys` had no auth check at all.

### Phase 6 — Command Platform (downlink)
`device_commands` table is the single source of truth (no Redis queue — a deliberate choice, since commands need scheduling + retry timing better served by Postgres-as-source-of-truth than a delay queue). Interval dispatcher (`offline-detector.js`-style) publishes due/retry-due commands over MQTT; devices ack via `devices/{id}/commands/ack` (idempotent — only `sent`-status rows update) or a REST fallback. Retry on missing ack up to `max_attempts`, then `expired`.

### Phase 7 — Rules Engine
Generic IF-conditions-THEN-actions engine (explicitly *not* a multi-step/branching workflow engine — clarified with the user upfront). Conditions use dot-path payload lookup (`getFieldValue`, later promoted to `src/shared/object-path.js` in Phase 9). Actions run through a pluggable registry (`webhook`, `mqtt_publish`, `device_command`, `notification`) — the `notification` action was a logging-only stub until Phase 8. Rule versioning (snapshot-before-update into `rule_versions`) and a side-effect-free `/test` dry-run endpoint.

### Phase 8 — Notification Platform
Real delivery behind the Rules Engine's `notification` action: Alerts (`alerts` table) with multi-level escalation policies (`[{delayMs, channels}]`, level 0 immediate, later levels advance automatically if unacknowledged), Notification Deliveries (`notification_deliveries`, one row per channel/recipient attempt — the actual retry queue, genuinely async unlike Rules' synchronous actions), Templates (`{{dot.path}}` rendering). Channels: `email` (real, via a new **Mailpit** dev container) and `webhook` (real) are fully functional; `sms`/`push` are documented stubs (no Twilio/FCM credentials) that always succeed since there's no real failure mode to simulate.

**Security fix found here**: the initial `nodemailer` install pulled a version with several real CVEs (SMTP command injection, CRLF injection) — pinned to the patched `9.0.3`.

### Phase 9 — Dashboard Platform (+ first frontend)
Backend `dashboards`/`dashboard_templates` module (JSON-configured widget `layout`, no versioning, plain hard-delete since nothing FK-references dashboards). Widget types: `chart`, `gauge`, `status_card`, `alarm_list`, `map` (device location comes from the existing flexible `devices.metadata.location`, no new column). `GET /:id/data` resolves every widget in one call, each individually try/caught.

First real frontend: React + Vite SPA in `frontend/`, JSON-configured dashboard editing (explicitly *not* drag-and-drop — clarified upfront), polling-based live updates (no websockets). Added `POST /api/v1/auth/refresh` (previously missing entirely) so the SPA doesn't force re-login every 15 minutes.

**Fixes found here**: a vulnerable `esbuild`/Vite dependency chain in the new frontend tooling (patched to Vite `6.4.3`, 0 vulnerabilities); a pre-existing CI gap where the backend test job had no Postgres/Redis/EMQX/Mailpit service containers or migration step at all — meaning `npm test` in CI had almost certainly been silently non-functional since Phase 5's "test against real infra" requirement began. Both fixed, but the CI change is **unverified** (no way to trigger an actual GitHub Actions run from this environment).

### Phase 10 — Reports & Analytics
New `reports`/`report_schedules`/`report_runs` tables. A Report is a saved query definition (device scope + a list of `{field, label, aggregation}` telemetry metrics + optional alert/command KPI blocks + `bucketInterval`/`periodDays`). Aggregation is real SQL (`date_trunc` grouping + `(payload #>> path::text[])::numeric` extraction on `telemetry_history`), not app-level math; **Trend Analysis** is just the same aggregate query run twice (current period vs. the immediately-preceding period of equal length) — no separate subsystem. KPI blocks (alert counts + avg acknowledge time by severity; command counts by status) reuse the existing `alerts`/`device_commands` tables, no new tracking.

**Export**: `pdfkit` (PDF: KPI lines + per-metric tables, no chart screenshots — keeps this backend headless-browser-free) and `exceljs` (Excel: one sheet per metric + a Summary sheet). Both are pure functions over the same JSON the `/data` endpoint returns, so the ad-hoc export route, the scheduled dispatcher, and tests all exercise identical code.

**Scheduled Reports**: frequency presets (`daily`/`weekly`/`monthly`) + a computed `next_run_at` (plain JS date math, no cron dependency), drained by a `setInterval` dispatcher identical in shape to the command/notification dispatchers. Files are never persisted (in-memory only, per an explicit scope decision) — scheduled runs email the export as an attachment (extended `sendEmail` to pass through `attachments`); `report_runs` records success/failure history without storing the file itself. MinIO remains unused, consistent with every prior phase.

Frontend gained `frontend/src/reports/` (list/view/edit/schedules pages), mirroring the Dashboards pages, plus a new `apiDownload`/`triggerBlobDownload` pair in the API client for binary file downloads (the existing `apiRequest` only handled JSON).

**Verified live, not just via tests**: created a report against real telemetry rows spanning multiple day-buckets and confirmed the returned aggregation/trend numbers by hand; downloaded and byte-verified a real PDF (`%PDF` header) and a real Excel file (`PK` header, parseable by `exceljs`); forced a schedule's `next_run_at` into the past and watched the live server's own dispatcher (not a test call) deliver a real email with a real attachment to Mailpit within its normal 60s poll interval; drove the new frontend pages in a real headless browser (login → list → view → download both formats → add a metric → add a schedule) with zero console errors.

**Accepted, non-blocking**: `exceljs`'s transitive `uuid` dependency has a moderate advisory (`GHSA-w5hq-g745-h8pq`, missing bounds check when a `buf` option is explicitly passed to `uuid.v3/v5/v6`); this codebase never calls `uuid` directly and exceljs doesn't expose that option, so the vulnerable code path isn't reachable. The only available fix (`npm audit fix --force`) downgrades exceljs to `3.4.0`, a major regression — not taken.

### Phase 11 — Firmware & Remote Management
Every deliverable (OTA, rollback, remote configuration, diagnostics, log collection, reboot) is a new `device_commands.type` riding the existing Command Platform (Phase 6) rather than a parallel downlink mechanism — no new MQTT topics, no changes to the mqtt module. New tables: `firmware_versions` (registry), `firmware_deployments` (thin join to `device_commands` — "current firmware" and deployment status are derived live via SQL join, never denormalized), `device_configurations` (desired/reported device-shadow pair, matching AWS IoT Device Shadow / Azure IoT Hub twin semantics), `log_collections` (request/upload lifecycle).

**Command ack hooks**: added a small pluggable registry (`registerAckHook(type, handler)` in `command-service.js`) — the same shape as the Rules Engine's action registry and the Notification Platform's channel registry, just applied to command acknowledgement. The firmware module registers one hook (`config_update` → writes `device_configurations.reported_config`/`reported_version`), letting it react to acks without the commands module importing it.

**This is the first phase to genuinely need file storage** — `src/shared/storage.js` was rewritten from a `putObject`-only stub onto the real `minio` npm package (`ensureBucket`/`putObject`/`getObjectStream`, one bucket, key-prefixed). Per an explicit user decision, devices never talk to MinIO directly: firmware downloads and log uploads are proxied through authenticated Express routes (`multer`, memoryStorage, size-capped) that stream to/from MinIO server-side — consistent with the fact that devices never talk directly to any other backing store in this platform, only MQTT and the Express API.

**Rollback** has no separate machinery: it finds the most recent deployment whose command is `acknowledged`, then the one before it, and redeploys that firmware. **Log collection** is modeled as two independent events (the `collect_logs` command prompting the device, and a separate upload endpoint the device hits with the actual bytes) since the command ack itself doesn't carry the file.

**No new RBAC permissions** — every action reuses an existing permission (`manage_devices` for firmware registry uploads, `send_commands` for anything that creates a command under the hood, `ack_commands` for device-initiated downloads/uploads, `read` for everything else), so `auth-service.js`'s roles map was untouched this phase.

Frontend gained `frontend/src/devices/` (a device list page — none existed before — plus a per-device management panel for firmware/config/diagnostics/logs/reboot) and `frontend/src/firmware/` (registry + upload form), plus a new `apiUpload` FormData helper in the API client.

**Bug found and fixed during manual verification**: `DeviceManagementPage.jsx` only guarded its loading state on `!device`, but `device` gets set (via a separate `setDevice` call) a render-tick before the other five parallel API calls (`configuration`, `firmware`, `diagnostics`, etc.) resolve — a real race that crashed with `Cannot read properties of null (reading 'desired_version')` on every page load. Caught by the Playwright verification pass (not by any automated test, since the race depends on real network timing), fixed by also guarding on `!configuration`.

**Verified live, not just via tests**: uploaded a real firmware file and confirmed the sha256 checksum computed server-side matched a local computation; downloaded it back through the device-facing route and diffed byte-for-byte (identical); deployed it to a real device, let the live dispatcher publish the MQTT command, acked it as the device, and confirmed "current firmware" correctly derived from the join; pushed a configuration and confirmed the ack hook updated `reported_config` on the live running server (not a test call); requested and uploaded a real log file, downloaded it back, diffed byte-for-byte (identical); drove the new frontend pages in a real headless browser end-to-end (deploy → ack via direct API calls → reload → current version updates; upload a second firmware version from the browser; rollback; correctly observed that "current" stays on the old version until the rollback's command is itself acknowledged — confirming the derived-state design works as intended, not a bug).

### Phase 12 — Plugin Framework
The charter's explicit purpose for this phase: industry-specific logic (agriculture, industrial automation, etc.) starts living **outside** the core codebase, plugged in rather than merged in. `src/plugins/` is the framework (loader, manifest validation, plugin-scoped API, a thin `#plugin-sdk` authoring package); `plugins/` (repo root) holds actual plugins, one directory each with a `plugin.json` manifest.

**In-process, not isolated** — plugins are trusted local Node modules, dynamically `import()`ed into the same process (via `pathToFileURL` for cross-platform-safe absolute-path imports, which matters on Windows). Same trust model as Fastify/ESLint/Babel plugins; no child-process/RPC layer, since this is a single-tenant deployment with operator-installed plugins, not a marketplace running untrusted code.

**Extension surface reuses (and extends) existing pluggable registries**: Rule actions and Notification channels already had a `{type: fn}` shape (Phases 7-8) — just needed `registerX`/`unregisterX` added. Dashboard widget resolution (Phase 9) was a hardcoded `switch` — refactored into the same registry shape. Every `registerX` throws on a name collision (built-in or another plugin) rather than silently overwriting.

**A necessary, deliberate validation change**: the `type`/`channel` fields on rule actions, notification channels, and dashboard widgets were Joi `.valid(...)` static whitelists — fundamentally incompatible with an open, plugin-extensible type namespace, since they're baked in at module load before any plugin could exist. Relaxed to `Joi.string().min(1)` in all three `validation.js` files; enforcement moved to where it already effectively happened (evaluation-time "Unknown type" errors, already-existing per-action try/catch in `rule-service.js` recording failures in `rule_history`).

**Lifecycle is 3 real states** (new `plugins` table): `active`, `disabled`, `error`. Auto-enable on first discovery; an operator's disable choice persists across restarts. Disabling is real: `createPluginApi` tracks every `registerX` call a plugin makes, and disabling walks that list calling the matching `unregisterX` for each (plus the plugin's own `deactivate()` hook) — a disabled plugin's action/channel/widget genuinely stop resolving, not just a DB flag. Plugin-owned Express routes (mounted at `/api/v1/plugin-extensions/:pluginName/*`, deliberately separate from the `/api/v1/plugins` management API) can't be truly unmounted, so each gets a small always-mounted gate middleware checking a live in-memory status flag, returning `503` when disabled.

**SDK vs. API are different surfaces**: the SDK (`#plugin-sdk`, a Node `package.json` `"imports"` subpath alias — no new dependency) is authoring sugar (`definePlugin` + JSDoc typedefs, since this project has no TypeScript). The API is the capability object passed to `activate(api)`: the three `registerX` methods, an Express `router`, a prefixed `logger`, `auth` (the real `authenticate`/`requirePermission`/`authenticateApiKey` middleware, so plugins don't need relative imports into `src/`), and curated helpers (`createDeviceCommand`, `getLatestTelemetry`, `getFieldValue`).

**Permission**: `manage_plugins` added to `admin` only — deliberately **not** `manager`, breaking the pattern every prior phase followed, because enabling/disabling arbitrary loaded code is materially more sensitive than the other `manage_*` permissions.

**Sample plugin**: `plugins/sample-agriculture-plugin/` — an `irrigation_command` rule action (sends a real device command) and a `soil_moisture` dashboard widget, explicitly labeled a reference implementation (its own `README.md` says so). Frontend gained `frontend/src/plugins/PluginListPage.jsx` (list + enable/disable toggle).

**Verified live, not just via tests**: confirmed the real server auto-discovers and activates the sample plugin on boot (log line, `GET /api/v1/plugins`); created a real rule using `irrigation_command`, published real telemetry, confirmed a real `irrigation` device command was created; created a real dashboard with a `soil_moisture` widget and confirmed it resolved real telemetry; hit the plugin's custom route; disabled the plugin live (no restart) and confirmed the route started 503ing AND the dashboard widget started returning `null` (real unregistration, not just a status flag) AND a fresh rule evaluation no longer created a command; re-enabled and confirmed everything came back — all against the live running process, then repeated via a real headless browser against the new Plugins admin page with zero console errors.

### Phase 13 — Production Hardening
Cross-cutting, not a new feature module: rate limiting, health checks, metrics/Prometheus/Grafana, backup/restore/DR, targeted performance fixes, container hardening, and a security review. Also fixed a real, pre-existing gap: CI's backend job never got a MinIO service container when Phase 11 made MinIO real infrastructure — `npm test` in CI had almost certainly been broken since then.

**Rate limiting was a live bug, not just under-hardened**: the single global 200 req/15min limiter was actually *too low* — a Phase 9 dashboard alone polls every 5s, consuming 180 of that budget from one browser tab. Split into a strict `authLimiter` (10/15min, login/register/refresh only — real brute-force protection) and a generous, env-tunable general limiter (default 2000/15min).

**Health checks split liveness vs. readiness**: `GET /health` (cheap, no I/O) vs. new `GET /health/ready` (real Postgres/Redis/MQTT/MinIO pings, each wrapped in a 3s timeout because ioredis doesn't fail fast on a dead connection by default — discovered by literally stopping the Redis container mid-verification and watching the first version hang for ~90 seconds).

**Metrics** (`prom-client`): default process/event-loop metrics plus an `http_request_duration_seconds` histogram. Getting the route label right took real debugging: Express resets `req.baseUrl` while walking back to the top-level error handler on any thrown error (the `catch (err) { next(err) }` pattern used in every route handler in this codebase), silently mislabeling a large fraction of error responses. Fixed by reconstructing the module prefix from the untouched `req.originalUrl` against a fixed list of mount prefixes instead of trusting `req.baseUrl`.

**Prometheus + Grafana**: `docker-compose.monitoring.yml`, optional, layered on the dev stack. Config-as-code (scrape config, datasource, and a real 6-panel dashboard all provisioned from files, not click-ops). Verified live: real scrape target `UP`, real PromQL query results with correctly-labeled routes, Grafana API confirming both the datasource and dashboard auto-provisioned.

**Performance**: `compression` middleware; explicit `pg.Pool` tuning (previously 100% library defaults); an honest index audit (every `CREATE INDEX` cross-referenced against every `REFERENCES` in `src/migrations/*.sql`) that found the schema already well-indexed, with exactly one real gap closed (`report_schedules.report_id`) — a genuine finding, not a failure to look hard enough; a real `.dockerignore` (previously absent — `COPY . .` could overwrite the container's freshly-installed `node_modules` with the host's own).

**Container hardening**: two-stage Dockerfile, non-root `node` user (previously ran as root), dependency-free `HEALTHCHECK`. Verified live: real `docker build`, confirmed `whoami` → `node`, confirmed `plugins/` ships while `tests/`/`frontend/`/`.git` don't, confirmed the container reports `healthy` via `docker inspect`.

**Backup/Restore/DR**: real `scripts/backup.sh`/`scripts/restore.sh` (`pg_dump -Fc`/`pg_restore --clean`) plus `docs/operations/disaster-recovery.md`. Verified end-to-end: a real backup, restored into a throwaway database, row counts diffed and matched exactly — then the throwaway database and backup file were cleaned up.

**Security review**: the repo's `security-review` skill requires a git diff against `origin/HEAD`, which doesn't exist here (zero commits, no remote) — a hard environmental blocker, not routed around by fabricating git state. Did a manual review instead, informed by exactly where this codebase's real risk surface is. Found and fixed:
- **The most serious finding, caught only during live manual verification, not static review**: `redisClient`/`mqttClient`/`dbClient` (`pg.Pool`) had no `'error'` event listener — Node's default behavior for an unlistened EventEmitter `'error'` event is to throw, meaning a routine Redis restart, MQTT reconnect, or Postgres blip could crash the *entire process*. Confirmed by stopping Redis and watching the server actually crash before the fix, then survive after.
- `pino-http` logs the raw `Authorization`/`x-api-key` header by default — every bearer token and API key was landing in stdout logs verbatim. Fixed with `redact`.
- Real SSRF exposure: the Rules Engine's `webhook` action and the Notification Platform's `webhook` channel both `fetch()` a fully user-supplied URL with no validation — an authenticated `manage_rules`/`manage_notifications` user (not just `admin`) could target internal-only services or cloud metadata endpoints. Fixed with a literal hostname/IP blocklist (`shared/url-safety.js`), env-gated bypass for this project's own local-test-server-based webhook tests.
- `Jwt.verify` didn't pin `algorithms: ['HS256']`; a plugin manifest's `main` field had no path-traversal check; firmware upload's `deviceType`/`version` built an unsanitized MinIO object key — all fixed, all low-to-moderate severity.
- Reviewed and accepted without a code change: `cors()`'s wildcard default (this API's Bearer-token, non-cookie auth model doesn't give it the same CSRF implications cookie-based sessions would have).

All fixes re-verified against the full test suite (4 full runs across the phase as fixes landed — every run stayed green, 21 suites / 95 tests) and against the live running server, not just re-reading the code.

### Phase 14 — Cloud Deployment
Everything before this phase assumed local dev — `docker-compose.dev.yml` publishes everything to the host for debugging, and the frontend's `nginx.conf` had carried a literal TODO for its backend upstream hostname since Phase 9. This phase makes a real production rollout possible: `docker-compose.prod.yml`, a TLS-terminating Nginx, Cloudflare Origin Certificate SSL, CI/CD image publishing to GHCR, and a health-check-gated deploy/rollback script.

**Six named cloud targets (generic VPS, DigitalOcean, Hetzner, AWS, Azure, GCP), one built path**: this project has never touched a cloud SDK and every prior phase's infra has been Docker-Compose-only — building six separate integrations would be a first-time departure from that precedent, with no real cloud account here to verify any of them against. Instead: one deeply-built "any Linux host with Docker + SSH" path, genuinely identical for a generic VPS/DigitalOcean/Hetzner, documented to apply the same way to AWS EC2/Azure VM/GCP Compute Engine; each platform's managed-container alternative (ECS/Container Apps/Cloud Run) is named as a future option in `docs/operations/cloud-deployment.md`, not built.

**One Nginx, not two**: the frontend's existing `nginx:alpine` image (Phase 9) becomes the single public-facing edge — TLS termination, static serving, and `/api/` proxying all in one hop — rather than adding a dedicated reverse-proxy layer in front of it. `frontend/nginx.conf` (baked into the image) stays plain-HTTP; a new `frontend/nginx.prod.conf` is mounted **over** it by `docker-compose.prod.yml` (a volume override, not a Dockerfile change).

**Cloudflare Origin Certificate, not Certbot**: Cloudflare's proxy handles the public-facing TLS/CDN/DDoS layer; Nginx uses a free, 15-year Cloudflare Origin Certificate for Full (Strict) mode — real TLS on both legs, no renewal automation needed. MQTT is explicitly not covered (Cloudflare's HTTP(S) proxy doesn't carry raw MQTT TCP) — EMQX's own TLS listener (`8883`) is documented as a recommended hardening step, not built.

**Images pull by default, build locally as a fallback**: each app service sets both `image:` (a GHCR tag) and `build:` — Compose uses `image` for normal `up`, but `docker compose build` still works without touching GHCR, useful before CI is set up.

**"Rolling Deployment" means health-check-gated recreate, not blue-green**: `scripts/deploy.sh` runs `docker compose up -d --wait`, blocking until the new container's Phase 13 `HEALTHCHECK` passes before the old one is removed. A few seconds of downtime during the swap, not zero-downtime — a true blue-green setup was deliberately not built, since every background dispatcher in this codebase assumes exactly one running instance, and running two briefly during a cutover would need each re-audited for that safety.

**Secrets stay `.env`-file-based** — `.env.production.example` + a new `scripts/generate-secrets.sh` (real, `openssl rand`-based) rather than integrating any cloud provider's native secret manager, consistent with staying infra-agnostic.

**A real, unplanned incident during verification**: bringing up `docker-compose.prod.yml` without an explicit Compose project name caused it to *replace* the already-running dev-stack containers — same directory, same default project-name derivation, same container names. `docker-compose.dev.yml`'s Postgres has no named volume, so this wasn't just "two stacks running side by side," it destroyed the dev containers' data outright (session test/verification data only, nothing the user had asked to preserve — but a real, disclosed side effect, not glossed over). Root-caused and fixed with an explicit `name: iot-platform-prod` in `docker-compose.prod.yml`; the dev stack was then rebuilt fresh and the full test suite re-run to confirm it — green again, 21 suites / 95 tests.

**Verified live**: a real locally-generated self-signed certificate stood in for a Cloudflare Origin Certificate (drop-in file swap, not a config change, to use a real one); real `docker compose build` + `up -d --wait` brought up all six services with genuine healthcheck gating; a real `/api/v1/auth/login` request through the TLS+Nginx layer returned a real `401` from the backend container (proving the Phase 9 TODO is genuinely resolved, not just documented as resolved); real HSTS/X-Content-Type-Options/X-Frame-Options/Referrer-Policy headers and the `:80`→`:443` redirect confirmed in actual response headers; the cert actually being served was confirmed via `openssl s_client`; `scripts/generate-secrets.sh` confirmed to print usable random values; the exact `docker compose pull`/`up -d --wait`/readiness-check mechanic `scripts/deploy.sh` wraps in SSH was run directly and confirmed to work — the SSH-transport wrapper itself, and the CI workflow, remain unverified against a real remote host/GitHub Actions run, same honest caveat as every prior CI change in this project.

## Cross-cutting architectural patterns

- **Module shape**: every module owns `routes.js`, `services/`, `repositories/`, `validation.js` (Joi). Routes stay thin (validate → call service → respond); business logic and error status codes (`Object.assign(new Error(...), {status})`) live in services.
- **Repositories**: `BaseRepository` (generic `findAll`/`findById`/`create`) is extended for simple tables; tables needing custom queries (composite lookups, upserts, status transitions) get hand-written raw-SQL repository classes instead.
- **pg jsonb array gotcha** (came up repeatedly — rules `conditions`/`actions`, notifications `escalation_policy`, dashboards `layout`): `pg` serializes JS arrays as Postgres array literals, not JSON, unlike plain objects (which get auto-`JSON.stringify`'d). Every repository writing an array into a jsonb column does `JSON.stringify()` explicitly.
- **Background dispatchers**: `offline-detector.js` (Phase 5), `commands/dispatcher.js` (Phase 6), `notifications/dispatcher.js` (Phase 8), `reports/dispatcher.js` (Phase 10) all share one shape — bare `setInterval`, `.unref()`'d, returns a stop function, started in `server.js` and stopped in its graceful-shutdown handler. Each also exports its due-row-processing function standalone from the `setInterval` wrapper, so tests call it directly against rows forced into the past rather than waiting on real wall-clock time. Phases 11-12 introduced no new dispatcher.
- **Pluggable registries** instead of hardcoded switches: Rules Engine actions, Notification channels, Dashboard widgets (refactored from a `switch` in Phase 12), and Command ack hooks (Phase 11) are all `{type: handlerFn}` maps with `registerX`/`unregisterX` pairs — adding (or, since Phase 12, removing) a type/hook doesn't touch the evaluation/dispatch/acknowledgement loop it plugs into. This exact pattern is what made the Plugin Framework's extension surface cheap to build — three of its four extension points were "add a register function to something that already had this shape."
- **RBAC**: a single flat `roles` map in `src/modules/auth/services/auth-service.js` (`admin`/`manager`/`viewer`/`device` → permission arrays). Every module through Phase 10 added its own `manage_*` permission here; Phase 11 reused existing permissions instead of inventing `manage_firmware`; Phase 12 added exactly one new permission (`manage_plugins`) but scoped it to `admin` only — the first permission in this codebase not also granted to `manager`.
- **Derived state over denormalization**: `device_commands.status` doubles as the command history AND the source of truth for anything downstream (Phase 11's "current firmware" and deployment status are computed via a live SQL join, never cached in a separate column) — consistent with `device_commands` already serving as both live dispatch queue and history since Phase 6.
- **Testing**: `tests/helpers/reset-db.js` (`TRUNCATE ... RESTART IDENTITY CASCADE`, updated every phase as tables were added) + `tests/helpers/auth.js` (`createAdminToken`/`createUserWithRoles`, real register+login+role-patch, not mocked). Integration tests exercise real MQTT round trips, real Mailpit-delivered email, real failing local HTTP servers for retry/failure paths, and (Phase 12) the real shipped sample plugin via the same discovery path the production server uses — no plugin fixtures or mocking.
- **Unlistened EventEmitter `'error'` events crash the process** (Phase 13 finding): applies to `pg.Pool`, `ioredis`, and `mqtt.js` clients alike — any shared client wrapping a Node EventEmitter that can emit `'error'` needs an explicit listener, or a transient infra hiccup becomes a full process crash. Worth checking for any *future* shared client added to `src/shared/`.
- **Manual verification catches things static review and automated tests can't**: two of Phase 13's most important findings (the EventEmitter crash bug, the metrics route-label bug) were found only by actually running the server and driving it for real — stopping a real container, reading real Prometheus output — not by reading the code or by the automated test suite (which, notably, never exercises the process-crash path at all, since Jest doesn't model "the whole process died"). Phase 14 added a third example: the Compose project-name collision was only discoverable by actually running `docker compose up` and watching it touch containers it shouldn't have.
- **Compose files sharing a directory need explicit, distinct project names** (Phase 14 finding): Docker Compose derives a project name from the directory alone when none is given, so any two compose files in the same directory (here: `docker-compose.dev.yml` and `docker-compose.prod.yml`) collide on container/volume/network names by default. `docker-compose.prod.yml` now sets `name: iot-platform-prod` explicitly; `docker-compose.dev.yml` and `docker-compose.monitoring.yml` don't have this problem in practice (the latter is additive, layered on top, not a competing full stack) but the lesson generalizes to any future compose file added to this repo.

## Known gaps / explicitly out of scope

- Per-device EMQX credentials/ACLs — backend connects as one trusted internal MQTT client; device-level broker auth is a broker-config concern, not built.
- Real SMS/Push delivery — stub providers, pending third-party credentials.
- Command cancellation, rule hard-delete, device-type-scoped rules, per-dashboard sharing/ACLs, dashboard versioning, drag-and-drop dashboard editing — not requested, not built.
- Multi-step/branching workflow orchestration — Rules Engine is intentionally single-pass IF/THEN.
- Automatic alerting from device-offline/command-expiry events — only Rules Engine and direct API calls create alerts today.
- Frontend automated tests (Vitest/RTL/Playwright) — verification every phase since 9 was manual (a scratch Playwright install, not part of the repo, used to drive the real dev servers in a headless browser).
- Production deployment topology — `frontend/nginx.conf`'s backend upstream hostname is a documented placeholder; no real docker-compose.prod.yml/K8s manifests exist yet.
- CI service-container fix (Phase 9) is still unverified — recommend watching the first real push/PR run. Phases 10-11 didn't touch CI (the generic `npm run migrate`/`npm test` steps already cover new migrations/tests as they're added).
- Report file persistence/archive (e.g. a MinIO-backed download history for past scheduled report runs) — explicitly deferred in Phase 10; `report_runs` records only status/metadata, not the file. (Firmware/logs, added in Phase 11, DO persist to MinIO — different tradeoff for a different kind of artifact.)
- Absolute (non-relative) report date ranges beyond the `/data`/`/export` query-param override — reports default to a rolling `periodDays` window; no saved "custom date range" report type.
- Firmware version deletion, presigned direct-to-MinIO device transfer (explicitly rejected in favor of backend proxying — see Phase 11 decisions), and any device-side OTA/agent implementation (out of scope — this is a platform, not firmware for a specific device family).
- Plugin-owned database schemas/migrations, plugin-owned background jobs, and child-process/worker isolation for plugins — all explicitly out of scope for Phase 12 per the locked-in extension-surface decision (Rule actions + Notification channels + Dashboard widgets + routes only).
- A true plugin marketplace/registry (remote install, versioned dependency resolution, signature verification) — this framework assumes an operator manually places a vetted plugin directory under `plugins/`, nothing more.
- Real device-type/dashboard-template plugin registration — the sample plugin covers actions/widgets/routes; extending device types or templates from a plugin wasn't part of the locked-in extension surface.
- DNS-rebinding-proof SSRF protection (Phase 13) — `shared/url-safety.js` checks the literal hostname/IP, not a DNS-resolved address. Documented, accepted residual gap.
- Multi-instance/horizontal scaling (Phase 13) — rate-limit counters and dispatcher state are in-process/in-memory; correct for one backend instance, not coordinated across replicas behind a load balancer. Out of scope for a single-tenant-per-deployment platform unless/until that assumption changes.
- MinIO backup/recovery (Phase 13) — `scripts/backup.sh` covers Postgres only; firmware/log files in MinIO are treated as lower-priority/regenerable, documented in `docs/operations/disaster-recovery.md` rather than backed up.
- Deep per-domain metrics (telemetry ingestion rate, command dispatch success rate, notification delivery latency, etc.) — Phase 13 scoped metrics to HTTP + process-level only, per the locked-in decision.
- Real per-provider IaC (Terraform/CloudFormation/ARM/Deployment Manager) and each cloud platform's managed-container service (ECS/Container Apps/Cloud Run) — Phase 14 named these as future options in `docs/operations/cloud-deployment.md`, deliberately not built (see that phase's locked-in scope decision).
- EMQX's own TLS listener (`8883`) for encrypting device MQTT traffic in production — documented as a recommended step in `docs/operations/cloud-deployment.md`, not configured by `docker-compose.prod.yml`.
- Auto-deploy-on-merge CI — `scripts/deploy.sh` is real but manually invoked; wiring it into a CI job needs real host/SSH secrets this environment doesn't have.
- DNS-rebinding-proof SSRF protection, multi-instance/horizontal scaling, MinIO backup/recovery — unchanged from Phase 13, still open (see that phase's entries above; Phase 14 didn't revisit them).
